🟢 Fase 1 — SELECT + WHERE Fundamento

O básico que todo mundo precisa dominar dormindo. Selecionar dados e filtrar. Parece simples, mas é 40% do trabalho diário de um analista.

1

Liste todos os serviços com preço acima de R$ 100

Fácil

A dona quer saber quais serviços são "premium" (acima de R$ 100). Mostre nome, categoria e preço, ordenados do mais caro ao mais barato.

Você vai precisar de: SELECT, FROM, WHERE e ORDER BY ... DESC

✅ Resposta

SELECT nome, categoria, preco FROM servicos WHERE preco > 100 ORDER BY preco DESC;

Explicação: WHERE filtra as linhas onde preço é maior que 100. ORDER BY com DESC coloca o maior primeiro. Simples e direto — essa é a base de qualquer consulta.

2

Clientes cadastrados em 2024

Fácil

Quantos clientes se cadastraram em 2024? Mostre nome, email e data_cadastro dos clientes cujo cadastro foi em 2024.

Use WHERE data_cadastro BETWEEN '2024-01-01' AND '2024-12-31' ou STRFTIME('%Y', data_cadastro) = '2024'

✅ Resposta

SELECT nome, email, data_cadastro FROM clientes WHERE data_cadastro BETWEEN '2024-01-01' AND '2024-12-31' ORDER BY data_cadastro;

Explicação: BETWEEN é inclusivo nos dois lados. Também funciona com STRFTIME ou LIKE '2024%'. Em entrevista, BETWEEN é mais limpo.

3

Funcionários ativos com salário acima de R$ 3.000

Fácil

RH quer a lista de funcionários ativos que ganham acima de 3 mil. Mostre nome, cargo, salario.

Duas condições no WHERE: ativo = 1 AND salario > 3000

✅ Resposta

SELECT nome, cargo, salario FROM funcionarios WHERE ativo = 1 AND salario > 3000 ORDER BY salario DESC;

Explicação: AND combina filtros. Ambas as condições precisam ser verdadeiras. É assim que o dia a dia funciona — raramente você filtra por uma coisa só.

4

Produtos com estoque abaixo de 10 unidades

Fácil

Alerta de reposição! Liste os produtos com estoque baixo. Mostre nome, marca, estoque e preco_venda, do menor estoque ao maior.

WHERE estoque < 10 ORDER BY estoque ASC

✅ Resposta

SELECT nome, marca, estoque, preco_venda FROM produtos WHERE estoque < 10 ORDER BY estoque ASC;

Explicação: ORDER BY ASC (ou sem ASC, que é o padrão) mostra do menor ao maior. Relatório clássico de estoque — gestores precisam disso toda semana.

5

Agendamentos cancelados ou faltados

Fácil

A gerente quer ver os agendamentos que não foram realizados. Mostre id, data_hora, status, valor_cobrado onde o status é 'cancelado' ou 'falta'.

Use WHERE status IN ('cancelado', 'falta') ou WHERE status = 'cancelado' OR status = 'falta'

✅ Resposta

SELECT id, data_hora, status, valor_cobrado FROM agendamentos WHERE status IN ('cancelado', 'falta') ORDER BY data_hora DESC;

Explicação: IN é mais limpo que vários OR. Funciona como uma lista de valores aceitos. Em entrevista, preferem ver IN — mostra que você é organizado.

6

Busca por nome parcial (LIKE)

Fácil

Uma cliente ligou mas só sabe que o nome começa com "Ma". Liste todos os clientes cujo nome começa com "Ma". Mostre nome, telefone, email.

WHERE nome LIKE 'Ma%' — o % substitui qualquer coisa depois

✅ Resposta

SELECT nome, telefone, email FROM clientes WHERE nome LIKE 'Ma%' ORDER BY nome;

Explicação: LIKE com % é o curinga do SQL. 'Ma%' = começa com Ma. '%silva%' = contém silva. '%son' = termina com son. Muito usado em buscas e relatórios.

🟡 Fase 2 — GROUP BY + Agregações Análise

Aqui é onde você deixa de ser "alguém que consulta dados" e vira "alguém que analisa dados". GROUP BY + funções de agregação = 30% do seu trabalho como analista BI.

7

Faturamento total do salão

Fácil

A dona perguntou: "quanto o salão já faturou no total?" Calcule o faturamento total (soma de valor_cobrado) dos agendamentos com status 'realizado'.

SELECT SUM(valor_cobrado) FROM agendamentos WHERE status = 'realizado'

✅ Resposta

SELECT SUM(valor_cobrado) AS faturamento_total, COUNT(*) AS total_atendimentos, ROUND(AVG(valor_cobrado), 2) AS ticket_medio FROM agendamentos WHERE status = 'realizado';

Explicação: SUM soma, COUNT conta linhas, AVG calcula média. O alias (AS) dá nome legível à coluna. ROUND arredonda decimais. Aproveitei pra já calcular ticket médio — analista bom já traz mais do que pedem.

8

Faturamento por mês

Médio

Relatório mensal. Mostre o faturamento agrupado por mês (formato YYYY-MM), número de atendimentos e ticket médio. Só agendamentos realizados.

Use STRFTIME('%Y-%m', data_hora) para extrair ano-mês. Agrupe por ele.

✅ Resposta

SELECT STRFTIME('%Y-%m', data_hora) AS mes, SUM(valor_cobrado) AS faturamento, COUNT(*) AS atendimentos, ROUND(AVG(valor_cobrado), 2) AS ticket_medio FROM agendamentos WHERE status = 'realizado' GROUP BY mes ORDER BY mes;

Explicação: STRFTIME extrai partes de datas no SQLite. GROUP BY agrupa todas as linhas do mesmo mês e aplica as funções de agregação em cada grupo. Esse é O relatório que todo gestor pede.

9

Top 5 serviços mais vendidos

Médio

Quais serviços são os campeões? Mostre os 5 serviços com mais agendamentos realizados, com nome do serviço, quantidade e faturamento total.

Você precisa de JOIN entre agendamentos e servicos. Agrupe por servico, ordene por COUNT DESC, e use LIMIT 5.

✅ Resposta

SELECT s.nome AS servico, COUNT(*) AS qtd_agendamentos, SUM(a.valor_cobrado) AS faturamento FROM agendamentos a JOIN servicos s ON a.servico_id = s.id WHERE a.status = 'realizado' GROUP BY s.nome ORDER BY qtd_agendamentos DESC LIMIT 5;

Explicação: JOIN conecta duas tabelas pela chave estrangeira. Alias (a, s) deixa o código mais curto. GROUP BY s.nome agrupa por serviço. LIMIT 5 = Top 5. Relatório tipo Pareto — quais poucos serviços geram maior receita.

10

Faturamento por categoria de serviço

Médio

Agrupe por categoria do serviço (ex: Cabelo, Unha, etc). Mostre categoria, total de atendimentos, faturamento e ticket médio.

JOIN com servicos, mas GROUP BY s.categoria em vez de s.nome.

✅ Resposta

SELECT s.categoria, COUNT(*) AS atendimentos, SUM(a.valor_cobrado) AS faturamento, ROUND(AVG(a.valor_cobrado), 2) AS ticket_medio FROM agendamentos a JOIN servicos s ON a.servico_id = s.id WHERE a.status = 'realizado' GROUP BY s.categoria ORDER BY faturamento DESC;

Explicação: Mesmo padrão do anterior, mas agrupando por categoria. Esse é o tipo de análise que vira gráfico de barras no Power BI.

11

Média de avaliação por funcionário

Médio

Qual funcionário é o melhor avaliado? Mostre nome do funcionário, média de nota, e quantidade de avaliações. Só quem tem pelo menos 3 avaliações.

Você precisa encadear 2 JOINs: avaliacoes → agendamentos → funcionarios. Use HAVING COUNT(*) >= 3 para filtrar depois do GROUP BY.

✅ Resposta

SELECT f.nome AS funcionario, ROUND(AVG(av.nota), 2) AS media_nota, COUNT(*) AS qtd_avaliacoes FROM avaliacoes av JOIN agendamentos a ON av.agendamento_id = a.id JOIN funcionarios f ON a.funcionario_id = f.id GROUP BY f.nome HAVING COUNT(*) >= 3 ORDER BY media_nota DESC;

Explicação: HAVING filtra DEPOIS da agregação (WHERE filtra ANTES). Dois JOINs encadeados conectam avaliações → agendamentos → funcionários. Isso é SQL intermediário — se fizer isso na entrevista, impressiona.

12

Faturamento por forma de pagamento

Fácil

Qual forma de pagamento gera mais receita? Mostre forma de pagamento, total e percentual do faturamento.

Para o percentual: divida o SUM de cada grupo pelo SUM total (subquery ou SUM() OVER()).

✅ Resposta

SELECT forma_pagamento, SUM(valor_cobrado) AS total, ROUND(SUM(valor_cobrado) * 100.0 / (SELECT SUM(valor_cobrado) FROM agendamentos WHERE status = 'realizado'), 1) AS percentual FROM agendamentos WHERE status = 'realizado' GROUP BY forma_pagamento ORDER BY total DESC;

Explicação: A subquery no SELECT calcula o total geral. Dividir o parcial pelo total e multiplicar por 100 dá o percentual. Relatório clássico financeiro.

🟠 Fase 3 — JOINs + Subqueries Intermediário

Agora ficou sério. JOINs são 25% do trabalho de analista de dados. Se você dominar isso, passa na frente de 80% dos candidatos Jr.

13

Clientes que nunca agendaram

Médio

Quais clientes se cadastraram mas nunca fizeram um agendamento? Mostre nome, email e data_cadastro. Isso é fundamental para campanhas de reativação.

Use LEFT JOIN + WHERE IS NULL, ou NOT IN com subquery, ou NOT EXISTS.

✅ Resposta

SELECT c.nome, c.email, c.data_cadastro FROM clientes c LEFT JOIN agendamentos a ON c.id = a.cliente_id WHERE a.id IS NULL ORDER BY c.data_cadastro;

Explicação: LEFT JOIN traz TODOS os clientes, mesmo sem agendamento. Onde não há correspondência, os campos de agendamentos ficam NULL. Filtrar WHERE a.id IS NULL pega exatamente quem nunca agendou. Padrão clássico — cai em toda entrevista.

14

Ranking de clientes por gasto total

Médio

Monte o Top 10 clientes que mais gastaram (soma de valor_cobrado). Mostre nome, quantidade de visitas e gasto total.

JOIN clientes com agendamentos, GROUP BY cliente, SUM(valor_cobrado), ORDER BY DESC, LIMIT 10.

✅ Resposta

SELECT c.nome, COUNT(*) AS visitas, SUM(a.valor_cobrado) AS gasto_total, ROUND(AVG(a.valor_cobrado), 2) AS ticket_medio FROM clientes c JOIN agendamentos a ON c.id = a.cliente_id WHERE a.status = 'realizado' GROUP BY c.nome ORDER BY gasto_total DESC LIMIT 10;

Explicação: Esse é o relatório de "melhores clientes" — toda empresa quer saber. É a base de programas de fidelidade. No Power BI você transforma isso em tabela matrix com drill-down.

15

Comissão de cada funcionário

Médio

Calcule a comissão total de cada funcionário: nome, total faturado, percentual de comissão e valor da comissão (faturado × comissao_pct / 100).

JOIN funcionarios com agendamentos. A comissão é SUM(valor_cobrado) * f.comissao_pct / 100

✅ Resposta

SELECT f.nome, f.comissao_pct, SUM(a.valor_cobrado) AS faturado, ROUND(SUM(a.valor_cobrado) * f.comissao_pct / 100.0, 2) AS comissao FROM funcionarios f JOIN agendamentos a ON f.id = a.funcionario_id WHERE a.status = 'realizado' GROUP BY f.nome, f.comissao_pct ORDER BY comissao DESC;

Explicação: Cálculo de comissão é um dos relatórios mais pedidos em salão/varejo. Note o 100.0 (float) para evitar divisão inteira. GROUP BY inclui comissao_pct porque é um campo não-agregado.

16

Clientes acima da média de gasto

Difícil

Liste clientes cujo gasto total é maior que a média geral de gasto por cliente. Mostre nome, gasto total, e quantas vezes visitou.

Subquery: calcule o gasto médio por cliente na subquery, e compare no HAVING. Ou use CTE.

✅ Resposta

SELECT c.nome, SUM(a.valor_cobrado) AS gasto_total, COUNT(*) AS visitas FROM clientes c JOIN agendamentos a ON c.id = a.cliente_id WHERE a.status = 'realizado' GROUP BY c.nome HAVING gasto_total > ( SELECT AVG(total) FROM ( SELECT SUM(valor_cobrado) AS total FROM agendamentos WHERE status = 'realizado' GROUP BY cliente_id ) ) ORDER BY gasto_total DESC;

Explicação: Subquery aninhada: primeiro calcula o total por cliente, depois calcula a média desses totais, e compara no HAVING. Esse padrão ("acima da média") é pergunta clássica de entrevista SQL.

17

Margem de lucro dos produtos

Médio

Calcule o lucro bruto de cada produto vendido: nome do produto, quantidade vendida, receita total, custo total e lucro (receita - custo). Ordene pelo mais lucrativo.

JOIN vendas_produtos com produtos. Receita = SUM(quantidade * valor_unitario). Custo = SUM(quantidade * preco_custo).

✅ Resposta

SELECT p.nome, SUM(vp.quantidade) AS qtd_vendida, ROUND(SUM(vp.quantidade * vp.valor_unitario), 2) AS receita, ROUND(SUM(vp.quantidade * p.preco_custo), 2) AS custo, ROUND(SUM(vp.quantidade * vp.valor_unitario) - SUM(vp.quantidade * p.preco_custo), 2) AS lucro FROM vendas_produtos vp JOIN produtos p ON vp.produto_id = p.id GROUP BY p.nome ORDER BY lucro DESC;

Explicação: Margem de lucro é um dos relatórios mais comuns em varejo/e-commerce. A lógica é: receita - custo = lucro bruto. Esse tipo de query vira gráfico de barras no dashboard.

18

DRE Simplificado do salão

Difícil

Monte um Demonstrativo de Resultado: Receita total de serviços, + Receita de produtos, - Total de despesas, - Total de salários = Resultado.

Use subqueries separadas para cada linha. SUM de agendamentos realizados, SUM de vendas_produtos, SUM de despesas, SUM de salários de funcionários.

✅ Resposta

SELECT (SELECT SUM(valor_cobrado) FROM agendamentos WHERE status = 'realizado') AS receita_servicos, (SELECT SUM(quantidade * valor_unitario) FROM vendas_produtos) AS receita_produtos, (SELECT SUM(valor) FROM despesas) AS despesas, (SELECT SUM(salario) FROM funcionarios WHERE ativo = 1) AS folha_salarial, (SELECT SUM(valor_cobrado) FROM agendamentos WHERE status = 'realizado') + (SELECT SUM(quantidade * valor_unitario) FROM vendas_produtos) - (SELECT SUM(valor) FROM despesas) - (SELECT SUM(salario) FROM funcionarios WHERE ativo = 1) AS resultado;

Explicação: DRE (Demonstrativo de Resultado do Exercício) é o relatório financeiro mais importante de qualquer negócio. Aqui usamos subqueries para calcular cada componente. Num projeto Power BI, cada linha vira um KPI card.

🔴 Fase 4 — CTEs + Window Functions Avançado

Se você dominar isso, já está acima de 90% dos candidatos Jr. CTEs + Window Functions são o que separam um analista que "sabe SQL" de um que realmente resolve problemas complexos. Esse é o 20% do esforço que gera 80% do resultado em entrevistas.

19

Variação MoM (Mês sobre Mês)

Difícil

Calcule o faturamento mensal com variação percentual em relação ao mês anterior. Mostre: mês, faturamento, faturamento anterior, variação %.

Use CTE para calcular faturamento mensal. Depois LAG(faturamento) OVER (ORDER BY mes) para pegar o mês anterior.

✅ Resposta

WITH mensal AS ( SELECT STRFTIME('%Y-%m', data_hora) AS mes, SUM(valor_cobrado) AS faturamento FROM agendamentos WHERE status = 'realizado' GROUP BY mes ) SELECT mes, faturamento, LAG(faturamento) OVER (ORDER BY mes) AS fat_anterior, ROUND( (faturamento - LAG(faturamento) OVER (ORDER BY mes)) * 100.0 / LAG(faturamento) OVER (ORDER BY mes), 1) AS variacao_pct FROM mensal ORDER BY mes;

Explicação: CTE (WITH) cria uma "tabela temporária" legível. LAG() é uma Window Function que pega o valor da linha anterior. Esse padrão é IDÊNTICO ao que você faria no Power BI com PREVIOUSMONTH(). Se fizer isso na entrevista, contrataram.

20

Ranking de funcionários por mês

Difícil

Para cada mês, faça um ranking dos funcionários por faturamento. Mostre: mês, nome, faturamento, posição no ranking.

RANK() OVER (PARTITION BY mes ORDER BY faturamento DESC). PARTITION BY cria "grupos" dentro da window function.

✅ Resposta

WITH fat_func AS ( SELECT STRFTIME('%Y-%m', a.data_hora) AS mes, f.nome, SUM(a.valor_cobrado) AS faturamento FROM agendamentos a JOIN funcionarios f ON a.funcionario_id = f.id WHERE a.status = 'realizado' GROUP BY mes, f.nome ) SELECT mes, nome, faturamento, RANK() OVER (PARTITION BY mes ORDER BY faturamento DESC) AS posicao FROM fat_func ORDER BY mes, posicao;

Explicação: PARTITION BY mes cria um "ranking separado para cada mês". Sem ele, o ranking seria geral. RANK() dá a mesma posição para empates (ex: dois 1ºs, próximo é 3º). DENSE_RANK() não pula números.

21

Acumulado (Running Total)

Difícil

Calcule o faturamento acumulado mês a mês. A dona quer ver o crescimento acumulado do negócio ao longo do tempo.

SUM() OVER (ORDER BY mes) — sem PARTITION BY, isso soma tudo desde o início até a linha atual.

✅ Resposta

WITH mensal AS ( SELECT STRFTIME('%Y-%m', data_hora) AS mes, SUM(valor_cobrado) AS faturamento FROM agendamentos WHERE status = 'realizado' GROUP BY mes ) SELECT mes, faturamento, SUM(faturamento) OVER (ORDER BY mes) AS acumulado FROM mensal ORDER BY mes;

Explicação: SUM() OVER (ORDER BY mes) sem PARTITION BY = running total. Cada linha soma tudo que veio antes + ela mesma. No Power BI, isso é TOTALYTD(). Gráfico de área com acumulado é um dos visuais mais bonitos.

22

Média Móvel 3 Meses

Difícil

Calcule a média móvel de 3 meses do faturamento. Isso suaviza variações e mostra a tendência real.

AVG(faturamento) OVER (ORDER BY mes ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)

✅ Resposta

WITH mensal AS ( SELECT STRFTIME('%Y-%m', data_hora) AS mes, SUM(valor_cobrado) AS faturamento FROM agendamentos WHERE status = 'realizado' GROUP BY mes ) SELECT mes, faturamento, ROUND(AVG(faturamento) OVER ( ORDER BY mes ROWS BETWEEN 2 PRECEDING AND CURRENT ROW ), 2) AS media_movel_3m FROM mensal ORDER BY mes;

Explicação: ROWS BETWEEN 2 PRECEDING AND CURRENT ROW cria uma "janela" de 3 linhas (2 anteriores + atual). AVG nessa janela = média móvel. Muito usado em finanças e BI para identificar tendências reais, ignorando picos pontuais.

23

% do total por categoria (Window)

Difícil

Para cada categoria de serviço, mostre faturamento e o percentual que ele representa do total geral — usando Window Function (não subquery).

SUM(faturamento) OVER () — sem ORDER BY nem PARTITION = soma de tudo. Divida o parcial por isso.

✅ Resposta

WITH cat AS ( SELECT s.categoria, SUM(a.valor_cobrado) AS faturamento FROM agendamentos a JOIN servicos s ON a.servico_id = s.id WHERE a.status = 'realizado' GROUP BY s.categoria ) SELECT categoria, faturamento, ROUND(faturamento * 100.0 / SUM(faturamento) OVER (), 1) AS pct_total FROM cat ORDER BY faturamento DESC;

Explicação: SUM() OVER () sem parâmetros = soma total de todas as linhas. Isso é o equivalente ao CALCULATE(ALL()) do DAX. Muito mais elegante que subquery. É o padrão que analistas sêniores usam.

24

Diferença entre vizinhos (LAG + LEAD)

Difícil

Para o faturamento mensal, mostre: mês, faturamento, faturamento do mês anterior (LAG), do mês seguinte (LEAD), e a diferença entre eles.

LAG() pega a linha anterior, LEAD() pega a próxima. Ambas aceitam ORDER BY.

✅ Resposta

WITH mensal AS ( SELECT STRFTIME('%Y-%m', data_hora) AS mes, SUM(valor_cobrado) AS fat FROM agendamentos WHERE status = 'realizado' GROUP BY mes ) SELECT mes, fat, LAG(fat) OVER (ORDER BY mes) AS anterior, LEAD(fat) OVER (ORDER BY mes) AS proximo, fat - LAG(fat) OVER (ORDER BY mes) AS diff_anterior FROM mensal ORDER BY mes;

Explicação: LAG(coluna, n) volta n linhas. LEAD(coluna, n) avança n linhas. Default é 1. Combinados, dão uma visão temporal completa — antes, agora e depois. Muito usado em análise de tendência.

25

Análise RFM (Recência, Frequência, Monetário)

Difícil

Segmente os clientes em RFM: para cada cliente, calcule quantos dias desde a última visita (R), quantas visitas (F) e quanto gastou no total (M). Classifique cada dimensão em 1-3 (usando NTILE ou CASE).

1) CTE base com JULIANDAY('now') - JULIANDAY(MAX(data_hora)), COUNT(*), SUM(valor_cobrado). 2) NTILE(3) OVER() para classificar em 3 grupos.

✅ Resposta

WITH rfm_base AS ( SELECT c.nome, CAST(JULIANDAY('now') - JULIANDAY(MAX(a.data_hora)) AS INTEGER) AS recencia, COUNT(*) AS frequencia, SUM(a.valor_cobrado) AS monetario FROM clientes c JOIN agendamentos a ON c.id = a.cliente_id WHERE a.status = 'realizado' GROUP BY c.nome ), rfm_score AS ( SELECT *, NTILE(3) OVER (ORDER BY recencia DESC) AS r_score, NTILE(3) OVER (ORDER BY frequencia ASC) AS f_score, NTILE(3) OVER (ORDER BY monetario ASC) AS m_score FROM rfm_base ) SELECT nome, recencia, frequencia, ROUND(monetario, 2) AS monetario, r_score, f_score, m_score, r_score || f_score || m_score AS rfm_segment FROM rfm_score ORDER BY monetario DESC;

Explicação: RFM é a segmentação de clientes mais usada no mercado. NTILE(3) divide automaticamente em 3 grupos iguais. Recência tem ORDER DESC porque menor = melhor (mais recente). Score 333 = melhor cliente. 111 = risco de churn.

26

Análise de horário de pico

Médio

Em qual dia da semana e horário o salão tem mais movimento? Monte uma análise cruzada dia × hora com contagem de agendamentos.

Use STRFTIME('%w', data_hora) para dia da semana (0=domingo) e STRFTIME('%H', data_hora) para hora.

✅ Resposta

SELECT CASE CAST(STRFTIME('%w', data_hora) AS INTEGER) WHEN 0 THEN 'Domingo' WHEN 1 THEN 'Segunda' WHEN 2 THEN 'Terça' WHEN 3 THEN 'Quarta' WHEN 4 THEN 'Quinta' WHEN 5 THEN 'Sexta' WHEN 6 THEN 'Sábado' END AS dia_semana, STRFTIME('%H:00', data_hora) AS hora, COUNT(*) AS agendamentos, SUM(valor_cobrado) AS faturamento FROM agendamentos WHERE status = 'realizado' GROUP BY dia_semana, hora ORDER BY agendamentos DESC LIMIT 15;

Explicação: CASE WHEN transforma números (0-6) em nomes legíveis. Agrupar por dia+hora mostra os picos de demanda. Isso é valioso para gestão: contratar mais nos picos, fazer promoções nos vales.

27

Cohort — Retenção de clientes por mês de cadastro

Difícil

Para cada mês de cadastro (cohort), quantos clientes fizeram agendamento naquele mês, no mês seguinte, 2 meses depois, etc? Análise de retenção.

Calcule o mês do cadastro e o mês de cada agendamento. A diferença em meses = período. GROUP BY cohort, periodo e conte clientes distintos.

✅ Resposta

WITH cohort AS ( SELECT c.id AS cliente_id, STRFTIME('%Y-%m', c.data_cadastro) AS mes_cadastro, STRFTIME('%Y-%m', a.data_hora) AS mes_visita, (CAST(STRFTIME('%Y', a.data_hora) AS INT) - CAST(STRFTIME('%Y', c.data_cadastro) AS INT)) * 12 + CAST(STRFTIME('%m', a.data_hora) AS INT) - CAST(STRFTIME('%m', c.data_cadastro) AS INT) AS periodo FROM clientes c JOIN agendamentos a ON c.id = a.cliente_id WHERE a.status = 'realizado' ) SELECT mes_cadastro, periodo, COUNT(DISTINCT cliente_id) AS clientes_ativos FROM cohort WHERE periodo >= 0 AND periodo <= 6 GROUP BY mes_cadastro, periodo ORDER BY mes_cadastro, periodo;

Explicação: Análise de cohort é uma das mais valorizadas no mercado. Mostra se o negócio retém clientes. Se o mês 0 tem 50 clientes e o mês 3 tem 10, a retenção é 20%. Coloque isso no portfolio e vai chamar atenção.

Análise ABC de serviços (Pareto)

Difícil

Classifique os serviços em A (top 80% da receita), B (80-95%) e C (últimos 5%) usando percentual acumulado.

1) CTE com faturamento por serviço. 2) % do total com SUM() OVER(). 3) SUM acumulado com ORDER BY fat DESC. 4) CASE para classificar.

✅ Resposta

WITH servico_fat AS ( SELECT s.nome, SUM(a.valor_cobrado) AS faturamento FROM agendamentos a JOIN servicos s ON a.servico_id = s.id WHERE a.status = 'realizado' GROUP BY s.nome ), servico_pct AS ( SELECT *, ROUND(faturamento * 100.0 / SUM(faturamento) OVER(), 1) AS pct, ROUND(SUM(faturamento) OVER (ORDER BY faturamento DESC) * 100.0 / SUM(faturamento) OVER(), 1) AS pct_acumulado FROM servico_fat ) SELECT nome, faturamento, pct, pct_acumulado, CASE WHEN pct_acumulado <= 80 THEN 'A' WHEN pct_acumulado <= 95 THEN 'B' ELSE 'C' END AS classe_abc FROM servico_pct ORDER BY faturamento DESC;

Explicação: Curva ABC (Pareto) mostra que poucos serviços geram a maioria da receita. É uma das análises mais valorizadas em BI. O acumulado com ORDER BY DESC garante que os maiores ficam no topo. Muito usado em gestão de estoque e serviços.

Dashboard SQL Completo — Resumo Executivo

Boss

Monte um resumo executivo completo em uma única query: faturamento total, ticket médio, total de clientes, média de avaliação, taxa de cancelamento (%), melhor mês e melhor funcionário.

Use múltiplas subqueries ou um CTE complexo. Cada métrica pode ser uma subquery no SELECT.

✅ Resposta

SELECT (SELECT ROUND(SUM(valor_cobrado), 2) FROM agendamentos WHERE status = 'realizado') AS faturamento_total, (SELECT ROUND(AVG(valor_cobrado), 2) FROM agendamentos WHERE status = 'realizado') AS ticket_medio, (SELECT COUNT(DISTINCT cliente_id) FROM agendamentos WHERE status = 'realizado') AS clientes_atendidos, (SELECT ROUND(AVG(nota), 2) FROM avaliacoes) AS media_avaliacao, (SELECT ROUND( COUNT(CASE WHEN status = 'cancelado' THEN 1 END) * 100.0 / COUNT(*), 1 ) FROM agendamentos) AS taxa_cancelamento_pct, (SELECT STRFTIME('%Y-%m', data_hora) FROM agendamentos WHERE status = 'realizado' GROUP BY STRFTIME('%Y-%m', data_hora) ORDER BY SUM(valor_cobrado) DESC LIMIT 1) AS melhor_mes, (SELECT f.nome FROM funcionarios f JOIN agendamentos a ON f.id = a.funcionario_id WHERE a.status = 'realizado' GROUP BY f.nome ORDER BY SUM(a.valor_cobrado) DESC LIMIT 1) AS melhor_funcionario;

Explicação: Cada subquery é independente. COUNT(CASE WHEN...) é uma forma elegante de contar condicionalmente. Esse resumo é exatamente o tipo de coisa que se coloca em KPI cards no Power BI. Uma query → 7 métricas.

Relatório Completo — Exportável para Dashboard

Boss

Monte a base de dados para um dashboard financeiro mensal: mês, receita serviços, receita produtos, despesas, lucro, variação MoM do lucro, acumulado do ano. Tudo em uma query.

Múltiplas CTEs: (1) receita_servicos por mês, (2) receita_produtos por mês, (3) despesas por mês. JOIN por mês, LAG para MoM, SUM() OVER() para acumulado.

✅ Resposta

WITH rec_serv AS ( SELECT STRFTIME('%Y-%m', data_hora) AS mes, SUM(valor_cobrado) AS receita_servicos FROM agendamentos WHERE status = 'realizado' GROUP BY mes ), rec_prod AS ( SELECT STRFTIME('%Y-%m', data_venda) AS mes, SUM(quantidade * valor_unitario) AS receita_produtos FROM vendas_produtos GROUP BY mes ), desp AS ( SELECT STRFTIME('%Y-%m', data) AS mes, SUM(valor) AS despesas FROM despesas GROUP BY mes ), consolidado AS ( SELECT rs.mes, COALESCE(rs.receita_servicos, 0) AS receita_servicos, COALESCE(rp.receita_produtos, 0) AS receita_produtos, COALESCE(d.despesas, 0) AS despesas, COALESCE(rs.receita_servicos, 0) + COALESCE(rp.receita_produtos, 0) - COALESCE(d.despesas, 0) AS lucro FROM rec_serv rs LEFT JOIN rec_prod rp ON rs.mes = rp.mes LEFT JOIN desp d ON rs.mes = d.mes ) SELECT mes, ROUND(receita_servicos, 2) AS rec_servicos, ROUND(receita_produtos, 2) AS rec_produtos, ROUND(despesas, 2) AS despesas, ROUND(lucro, 2) AS lucro, ROUND((lucro - LAG(lucro) OVER (ORDER BY mes)) * 100.0 / LAG(lucro) OVER (ORDER BY mes), 1) AS var_mom_pct, ROUND(SUM(lucro) OVER (ORDER BY mes), 2) AS lucro_acumulado FROM consolidado ORDER BY mes;

Explicação: Isso é exatamente o que um analista BI faz no dia a dia: consolida múltiplas fontes (serviços, produtos, despesas), calcula lucro, adiciona variação temporal e acumulado. Essa query é a base de um dashboard financeiro completo. Se você leva isso para a entrevista, acabou.
